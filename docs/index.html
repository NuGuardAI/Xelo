What is Xelo

A Bill of Materials built for AI

Software Bill of Materials (SBOM) tools were designed for packages and libraries. AI applications are different — they have agents, models, prompts, tools, and datastores that package managers cannot see.

Xelo fills that gap. It analyses your Python, Jupyter, and JavaScript/TypeScript source code, infrastructure files, SQL schemas, and Dockerfiles to produce an AI SBOM — a structured graph of every AI component it can find, complete with evidence, confidence scores, and relationship edges.

The output is a JSON document that you can validate, version-control, feed into your vulnerability scanner, export to CycloneDX, or query with your own tooling. It answers questions security teams actually need to ask: which agents touch PHI? which models are external providers? which datastores have no guardrail?

3-phase extraction pipeline
1

AST-aware adapters

Language-specific parsers for Python (ast) and TypeScript (tree-sitter) extract framework-specific components with high precision. No generic string matching.

2

Regex fallbacks

Model names, auth keywords, secret patterns, datastores, and IaC signals are caught across all file types — SQL, YAML, Dockerfiles, Nginx configs, prompt files.

3

LLM enrichment (optional)

An optional pass verifies uncertain detections, enriches node descriptions, generates a use-case summary, and produces a security briefing of IaC findings. Token budget controlled.

What gets detected

Every AI component, fully mapped

🤖

Agents

Agentic orchestrators — LangGraph graphs, CrewAI crews, AutoGen agents, OpenAI Agents SDK, Google ADK, AWS Bedrock, Azure Semantic Kernel. Captures graph topology and tool wiring.

🧠

Models

LLM and embedding model references with provider, version, and whether they are external services. Flags PHI flowing to third-party APIs.

🔧

Tools

Function tools, MCP tools, and decorated callables wired to agents. Includes privilege scope — db_write, code_execution, …

📝

Prompts

System instructions and prompt templates, including full content, injection risk score, and template variables that could be user-controlled.

🗄️

Datastores

Vector stores, databases, and caches. PII / PHI classification from SQL schema analysis and Python ORM models. Datastore type and transport captured.

🛡️

Guardrails

Content filters and safety validators. Used by the vulnerability scanner to identify AI components operating without any output protection.

🔑

Auth

Authentication nodes — OAuth2, API key, Bearer tokens, JWT, MCP auth providers. Surfaces API surface with insufficient authentication coverage.

⚡

Deployment & IaC

Docker image references, Kubernetes manifests, Terraform, CloudFormation, GitHub Actions workflows, Nginx configs. Cloud region, HA mode, OIDC, secret stores.

Quickstart

From install to AI SBOM in 60 seconds

Xelo's first two phases are fully deterministic and require LLM key. LLM enrichment is optional and token-budgeted. Xelo can be used in fully offline environments with no loss of core detection capabilities.

The JSON output can be validated with the bundled schema, exported as CycloneDX 1.6, or analysed with any of the built-in toolbox plugins.

Full guide → 
# Install
$ pip install xelo

# Scan a local repository
$ xelo scan ./my-ai-app --output sbom.json
  14 nodes, 18 edges → sbom.json

# Scan a remote repository
$ xelo scan https://github.com/org/repo --ref main

# Validate the output
$ xelo validate sbom.json
  OK — document is valid

# Run structural security rules
$ xelo plugin run vulnerability sbom.json

# Export to CycloneDX 1.6
$ xelo scan ./my-ai-app --format cyclonedx --output bom.cdx.json
from pathlib import Path
from xelo import AiSbomConfig, AiSbomExtractor, AiSbomSerializer
from xelo.toolbox.plugins.vulnerability import VulnerabilityScannerPlugin

# Extract
config = AiSbomConfig()
doc    = AiSbomExtractor().extract_from_path(
    path=Path("./my-ai-app"),
    config=config,
)
print(f"nodes={len(doc.nodes)}, edges={len(doc.edges)}")

# Serialize
json_str = AiSbomSerializer.to_json(doc)

# Vulnerability scan
sbom    = doc.model_dump(mode="json")
result  = VulnerabilityScannerPlugin().run(sbom, {})
print(result.status, result.message)
# .github/workflows/sbom.yml
name: AI SBOM scan
on: [push, pull_request]

jobs:
  xelo:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install xelo
        run: pip install xelo
      - name: Scan
        run: xelo scan . --output sbom.json
      - name: Security check
        run: |
          xelo plugin run vulnerability sbom.json \
            --output findings.json
          # fail if CRITICAL or HIGH findings present
          python3 -c "
          import json, sys
          d = json.load(open('findings.json'))
          s = d['details']['summary']
          sys.exit(1 if s['critical']+s['high'] > 0 else 0)
          "
      - name: Upload SARIF
        run: |
          xelo plugin run sarif sbom.json --output results.sarif
          xelo plugin run ghas  sbom.json \
            --config token=${{ secrets.GITHUB_TOKEN }} \
            --config github_repo=${{ github.repository }}
Toolbox plugins

Analyse, export, and integrate

Run any plugin with xelo plugin run <name> sbom.json. All offline plugins work with no network access and no API key.

vulnerability 21 structural VLA rules — missing guardrails, over-privileged agents, PHI to external LLMs, IaC security posture.
These rules extract the relevant controls from OWASP AI Top 10, NIST AI RMF, HIPAA, PCI-DSS and morph them for AI Applications. These rules can fire purely on the AI SBOM without a need for manual attestation or LLM enrichment, making them ideal for CI pipelines and offline environments. Offline
atlas Maps every VLA finding to MITRE ATLAS v2 techniques and recommended mitigations. Offline
sarif Exports all findings as SARIF 2.1.0 — compatible with GitHub Code Scanning and any SARIF viewer. Offline
cyclonedx Exports AI nodes as a standards-compliant CycloneDX 1.6 BOM for downstream supply-chain tooling. Offline
markdown Human-readable Markdown report covering all detected components, findings, and remediation guidance. Offline
license Checks all detected package dependencies against SPDX licence identifiers for compliance. Offline
dependency Scores dependency freshness and flags outdated or unmaintained AI packages in the dependency graph. Offline
ghas Uploads SARIF findings directly to GitHub Advanced Security — connects to GitHub API. Network
aws-security-hub Pushes findings to AWS Security Hub as Security Finding Format (ASFF) records. Network
xray Submits the SBOM to JFrog Xray for vulnerability enrichment and policy evaluation. Network
